# Linker Specification (linking EAST1 / EAST2 / EAST3)

This document defines the `linker` (link stage) specification when `EAST1` / `EAST2` / `EAST3` are handled as intermediate files.

Its two main goals are:

1. to allow each stage of the transformation pipeline to be verified independently
2. to finalize in one place information that cannot be fixed without looking at all modules, such as `type_id`

Positioning:

- The higher-level specification is [spec-east.md](./spec-east.md), and this document defines the details of its link stage as a lower-level specification.
- If there is a conflict, `spec-east` takes precedence, and this document must be updated to follow it.

## 1. Background

- After the introduction of `EAST3`, backends are expected to focus in principle on instruction mapping.
- On the other hand, assigning `type_id` requires cross-module information.
- In single-shot conversion, execution-order-dependent IDs are likely to appear, which harms reproducibility and multi-stage conversion.

Therefore, this document defines the responsibility split `EAST3 -> linker -> backend`.

## 2. Non-goals

- Replacing native binary linkers such as `ld` or `link.exe`
- Fully implementing optimizers such as DCE, inlining, or register allocation
- Abolishing the existing fast path (memory-only end-to-end pipeline)

## 3. Terms

- `EAST1`: the loss-minimal IR immediately after parsing
- `EAST2`: the IR dedicated to normalization
- `EAST3`: the semantic IR used as backend input
- `link unit`: one module’s `EAST3` document before linking
- `LinkedProgram`: a validated in-memory model that bundles multiple `link unit`s and program-wide options
- `link-input.v1`: the input manifest used to build a `LinkedProgram`
- `link-output.v1`: the output manifest that records the global summary and the output paths of linked modules
- `linked module`: an `EAST3` document after the linker / linked-program optimizer; it keeps `kind=Module` and `east_stage=3` while adding `meta.linked_program_v1`
- `helper module`: a synthetic `linked module` generated by the linked-program optimizer; it must carry `meta.synthetic_helper_v1` and is passed to the program artifact with `module_kind=helper`

## 4. Basic pipeline

### 4.1 Default (fast path)

- By default, processing remains in-memory as before.
- `parser -> EAST1 -> EAST2 -> EAST3(raw module) -> LinkedProgramLoader -> LinkedProgramOptimizer -> linked module(EAST3) -> backend`

### 4.2 Debug / reproducibility mode

- Save raw `EAST3` modules and `link-input.v1` / `link-output.v1` only when needed.
- The recommended route is:
  1. use `pytra-cli.py` to output raw `EAST3` modules and `link-input.json`
  2. use `pytra link` to read `link-input.json` and output `manifest.json` plus linked modules
  3. use `east2cpp.py` (for C++) or `east2x.py` (generic) to read `manifest.json` and emit through the backend

Recommended file names / extensions:

- `*.east1.json`
- `*.east2.json`
- `*.east3.json`
- `link-input.json`
- `manifest.json`

## 5. Responsibilities of the linker

The linker / linked-program optimizer receives a set of raw `EAST3` modules and finalizes the following:

1. Module-set validation
   - validate `kind=Module`
   - validate `east_stage=3`
   - validate `schema_version`
   - validate `meta.dispatch_mode`
   - fail fast on violations of program-wide consistency

2. Module-ID normalization and deterministic ordering
   - normalize drift in `module_id` and detect duplicate definitions
   - construct `LinkedProgram` in the same order for the same input set

3. Global summary construction
   - program-wide call graph
   - SCCs
   - `type_id_table`
   - non-escape summary
   - container ownership hints
   - If runtime-helper `@template` v1 is implemented, the canonical owner of the specialization collector / monomorphization is also the linker. Seeds must be determined from `FunctionDef.meta.template_v1` and call-site concrete type tuples, and must not fall back to raw decorators or be reanalyzed on the backend side.

4. Materialization into linked modules
   - materialize the necessary slice of the global summary into each module’s `meta.linked_program_v1` so that the backend can read module by module
   - function-level / call-level summaries (for example `FunctionDef.meta.escape_summary`, `Call.meta.non_escape_callsite`) may also be finalized at this stage
   - If optimizer-generated helpers are needed, synthetic helper modules may be added as long as they still keep `kind=Module` / `east_stage=3`. The canonical owner of helpers is the linker, and the backend must not rediscover or regenerate them.

5. Program-manifest output
   - output `link-output.v1` as the source of truth and the canonical source of global tables
   - the backend / `ProgramWriter` operates from `link-output.v1` plus linked modules

Backend-side prohibitions:

- do not recompute `type_id`
- do not reload module sets and rebuild whole-program summaries on their own
- do not supplement missing global information from raw `EAST3` in emitters or hooks
- do not rediscover or regenerate runtime-helper `@template` v1 specializations in the backend / `ProgramWriter`

## 6. `type_id` assignment rules

### 6.1 Basics

- built-ins:
  - adopt existing runtime fixed values as-is (for example `NONE=0`, `OBJECT=8`, `USER_BASE=1000`)
- user classes:
  - the linker assigns IDs greater than or equal to `USER_BASE`

### 6.2 Determinism (mandatory)

- For the same input set and the same options, the same `type_id` values must always be produced.
- The recommended assignment order is:
  1. topological order that satisfies inheritance dependencies
  2. lexical order of FQCN for ties

### 6.3 Validation

- If an undefined base type, inheritance cycle, or duplicate-name collision is detected, stop with `input_invalid`.

## 7. Input / output contract

### 7.1 Preconditions for raw `EAST3` documents

A raw `EAST3` document accepted by the linker must contain at least:

- `kind = "Module"`
- `east_stage = 3` (integer)
- `schema_version` (integer)
- `meta.dispatch_mode = "native" | "type_id"`
- `meta.transpiler_version` (recommended)
- `meta.input_hash` (optional but recommended)

### 7.2 `link-input.v1`

`link-input.v1` is the input manifest before `LinkedProgram` construction.

Required top-level keys:

- `schema`
  - fixed value: `pytra.link_input.v1`
- `target`
  - for example `cpp`, `rs`, `js`
- `dispatch_mode`
  - `native | type_id`
- `entry_modules`
  - array of entry-module `module_id`s
- `modules`
  - array of module entries

Optional top-level keys:

- `options`
  - an object holding target-specific / optimizer-specific options; unknown keys may be preserved transparently by the validator rather than fail-closed

Required keys in `modules[*]`:

- `module_id`
- `path`
- `source_path`
- `is_entry`

Path contract of `link-input.v1`:

- `path` and `source_path` use POSIX-relative paths from the manifest directory as the source of truth.
- Loader implementations may accept absolute paths for compatibility, but generators must output relative paths canonically.

Validation rules for `link-input.v1`:

1. `module_id` must be unique.
2. `entry_modules` must be a subset of `modules[*].module_id`.
3. Each `path` must point to a raw `EAST3` document with `kind=Module` / `east_stage=3`.
4. raw `EAST3.meta.dispatch_mode` must match manifest `dispatch_mode`.
5. The validator must normalize traversal order to lexical order of `module_id` to guarantee determinism.

### 7.3 `link-output.v1`

`link-output.v1` is the canonical output manifest of the linker / linked-program optimizer.

Required top-level keys:

- `schema`
  - fixed value: `pytra.link_output.v1`
- `target`
- `dispatch_mode`
- `entry_modules`
- `modules`
- `global`
- `diagnostics`

Required keys in `modules[*]`:

- `module_id`
- `input`
- `output`
- `source_path`
- `is_entry`
- `module_kind`
  - `user | runtime | helper`

Helper-only required keys in `modules[*]`:

- `helper_id`
- `owner_module_id`
- `generated_by`
  - fixed to `linked_optimizer` in v1

Required keys in `global`:

- `type_id_table`
  - `FQCN -> int`
- `call_graph`
  - `caller -> callee[]`
- `sccs`
  - `string[][]`
- `non_escape_summary`
  - program-wide summary object
- `container_ownership_hints_v1`
  - program-wide ownership-hint object

Required keys in `diagnostics`:

- `warnings`
  - array of strings
- `errors`
  - array of strings

Rules for `link-output.v1`:

1. `modules[*].output` points to the output path of the linked module (defined below).
2. All tables in `global` are required even if empty.
3. The canonical source of global tables referenced by the backend / `ProgramWriter` is always `link-output.v1`.
4. Entries with `modules[*].module_kind=helper` represent synthetic helper modules and must have `helper_id` / `owner_module_id` / `generated_by`.
5. Entries with `modules[*].module_kind=user|runtime` must not have `helper_id` / `owner_module_id` / `generated_by`.

### 7.4 Linked module schema

Like raw `EAST3`, linked modules keep `kind=Module` / `east_stage=3`.  
The added canonical metadata is `meta.linked_program_v1`.

Required keys in `meta.linked_program_v1`:

- `program_id`
  - a deterministic ID of the linked program; its generation rule is implementation-defined, but it must be identical for the same input set
- `module_id`
  - the current module’s `module_id`
- `entry_modules`
  - array of entry-module `module_id`s in the program
- `type_id_resolved_v1`
  - the `FQCN -> int` slice needed by the current module
- `non_escape_summary`
  - the escape-summary slice needed by the current module
- `container_ownership_hints_v1`
  - the ownership-hint slice needed by the current module

Supplemental rules for linked modules:

1. `meta.dispatch_mode` remains mandatory just as in raw `EAST3`, and must not conflict with `meta.linked_program_v1`.
2. Function-level / call-level materialized summaries may continue to use existing `meta` contracts.
3. `meta.linked_program_v1` is mandatory in linked modules and must not exist in raw `EAST3`.
4. Even when runtime-helper `@template` v1 is used, canonical function-level metadata remains in `FunctionDef.meta.template_v1`. The seed/source of truth of materialized specializations must not be pushed back to raw decorators.
5. Function clones materialized as implicit specializations may carry `FunctionDef.meta.template_specialization_v1`, and the program-wide summary may aggregate them into `link-output.global.runtime_template_specializations_v1`.

### 7.5 Synthetic helper module schema

Synthetic helper modules also keep `kind=Module` / `east_stage=3`, just like raw/linked modules. The added canonical metadata is `meta.synthetic_helper_v1`.

Required keys in `meta.synthetic_helper_v1`:

- `schema_version`
  - fixed value `1`
- `helper_id`
  - stable helper ID shared across backends
- `owner_module_id`
  - logical owner module ID of the helper
- `generated_by`
  - fixed value `linked_optimizer` in v1

Rules for synthetic helper modules:

1. A `link-output.modules[*]` entry with `module_kind=helper` must point to exactly one corresponding linked helper module.
2. `meta.linked_program_v1.module_id` points to the helper module’s own `module_id` and must not be confused with `meta.synthetic_helper_v1.owner_module_id`.
3. Since helper modules have no user source file, `source_path` may canonically be the empty string.
4. The backend / `ProgramWriter` must treat helpers using `meta.synthetic_helper_v1` as the source of truth, and must not rediscover helper/runtime/inline helper logic.

## 8. CLI / route specification (policy)

The canonical route assumes:

- `pytra-cli.py`
  - can output raw `EAST3` modules and `link-input.json`
- `pytra link`
  - reads `link-input.json` and outputs `manifest.json` plus linked modules
- `east2cpp.py`
  - accepts `manifest.json` and generates C++ multi-file output as an independent entrypoint that does not import non-C++ backends
- `east2x.py`
  - accepts either a raw single `Module` or `manifest.json`, and hands it to the backend through a generic route supporting all backends

Minimum behavior rules:

1. `--link-only` does not perform backend generation and outputs only `manifest.json` and linked modules.
2. `--object-dispatch-mode` is fixed before raw `EAST3` construction; the linker only performs consistency checks.
3. Even in debug / restart routes, `link-input.v1` / `link-output.v1` are treated as the canonical source.
4. Global passes use only the module set enumerated in `link-input.v1` / `link-output.v1` as input. They must not extend closure by following `source_path` or reparsing import statements.
5. The only closure `NonEscapeInterproceduralPass` may use on the linked-program route is `meta.non_escape_import_closure` filled by the linker/materializer. If it does not exist, it must be treated fail-closed as unresolved.
6. Implicit specializations for runtime-helper `@template` v1 must also complete entirely inside that enumerated module set. The specialization collector must create deterministic seeds from the call graph and resolved concrete type tuples, and must not implicitly expand into unlisted modules or all user code.

## 9. Implementation-mode guidance

### 9.1 Daily operation

- The default is in-memory `LinkedProgramLoader + LinkedProgramOptimizer`.
- Do not write raw `EAST3`, linked modules, or manifests on every run (prioritize speed).

### 9.2 Debug / CI

- Save raw `EAST3` dumps and `link-input.v1` / `link-output.v1` only when reproducing failures.
- In regression tests, make it possible to verify `link-input.v1 -> pytra link -> link-output.v1` independently.

## 10. Error contract

Linker failures are unified under `input_invalid(kind=..., stage=link, ...)`.

Recommended `kind` values:

- `type_cycle`
- `unknown_base_type`
- `duplicate_type_symbol`
- `duplicate_module_id`
- `missing_entry_module`
- `dispatch_mode_mismatch`
- `invalid_link_input`
- `invalid_link_output`
- `invalid_linked_module_meta`

## 11. Acceptance criteria

1. `type_id` assignment is deterministic for the same input.
2. For the same input set and the same options, the contents of `link-input.v1`, `link-output.v1`, and linked modules are deterministic.
3. Restarting through raw `EAST3` files yields the same result as the direct in-memory pipeline.
4. The linker detects mixed-`dispatch_mode` inputs and stops.
5. The backend does not recompute `type_id` or non-escape summaries and only consumes the values finalized by the linker.
